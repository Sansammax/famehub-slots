# Project Analysis: Training Center Booking System

## Overall Architecture
The application is a Single Page Application (SPA) built with React and Vite. It utilizes a modern frontend stack paired with a Backend-as-a-Service (BaaS) for data persistence, authentication, and edge computing.

### Tech Stack
* **Frontend:** React 19, TypeScript, Vite.
* **Styling & UI:** Tailwind CSS, Radix UI (via shadcn/ui).
* **Routing:** TanStack Router (`@tanstack/react-router`) for type-safe file-based routing.
* **Backend / Database:** Supabase (PostgreSQL, Authentication, Edge Functions).
* **Email Service:** Resend API (invoked via Supabase Edge Functions).

## Flow of Data
The data flow is primarily straightforward, moving from the client directly to the Supabase database, supplemented by edge functions for side-effects like emails.

1. **User Booking Flow:**
   * The user navigates to `/book` (`src/pages/book.tsx`). The component fetches existing public bookings (via an RPC call) and blocked dates from Supabase to render the `BookingCalendar`.
   * The user selects an available date, opening a multi-step slide-out form (using `Sheet`).
   * The user inputs their details. The form is validated locally using Zod.
   * Upon confirmation, the app directly calls `supabase.from("bookings").insert(...)`.
   * If the insert is successful, the client fires a non-blocking request to the `send-booking-email` Supabase Edge Function with the booking payload.
   * The Edge Function constructs HTML templates and sends emails via the Resend API to both the user and the admin.

2. **Admin Flow:**
   * Admins access `/admin/login`. They authenticate via Supabase Auth.
   * Upon successful login, they are redirected to `/admin/`. The `AdminDashboard` component mounts and verifies their admin status by checking the `user_roles` table.
   * The dashboard loads all bookings and blocked dates. Admins can view the bookings list, delete records, and toggle specific dates as "blocked" to prevent future bookings.

## Directory & File Breakdown

### 1. `src/` (Root Source)
Contains the core frontend application.
* **`main.tsx`**: The entry point. It creates the React root, configures the `RouterProvider`, and mounts the application inside the `index.html` shell.
* **`router.tsx`**: Initializes the TanStack router instance and defines global error handling components.
* **`routeTree.gen.ts`**: Auto-generated by the TanStack Router Vite plugin. It maps the file structure in `src/pages/` to route definitions.

### 2. `src/pages/`
Contains the file-based routes for the application.
* **`__root.tsx`**: The global layout wrapper. It provides the global `Outlet` for child routes and global UI elements like the `Toaster` for notifications.
* **`index.tsx`**: The landing page. Contains static promotional content, statistics, trainer profiles, and calls-to-action directing users to the booking page.
* **`book.tsx`**: The core booking application logic. It manages complex local state (calendar cursor, multi-step form progress, Zod form validation). It connects to Supabase to fetch availability and submit new bookings.
* **`admin/login.tsx`**: The authentication screen for administrators. It interfaces with `supabase.auth`.
* **`admin/index.tsx`**: The admin dashboard. It contains tabbed navigation (Dashboard, Bookings, Calendar, Settings). It handles fetching all bookings, deleting records, and inserting/deleting from the `blocked_dates` table.

### 3. `src/components/`
Contains reusable UI elements.
* **`booking/BookingCalendar.tsx`**: A highly interactive, custom calendar component. It calculates the month grid and visually distinguishes dates based on their state (`available`, `booked`, `blocked`, `past`). It accepts a `BookingMap` and blocked dates set as props to determine tile styles.
* **`site/`**: Contains layout components like `Navbar.tsx`, `Footer.tsx`, and the `Logo.tsx` branding.
* **`ui/`**: Contains atomic UI components generated by shadcn/ui (e.g., `button.tsx`, `input.tsx`, `sheet.tsx`). These rely heavily on Radix UI primitives and Tailwind for styling.

### 4. `src/services/`
Contains backend connectivity and configuration.
* **`supabase/client.ts`**: Initializes and exports the singleton Supabase JavaScript client, ensuring only one instance connects to the backend across the frontend app.
* **`supabase/types.ts`**: Contains the TypeScript interfaces representing the Supabase database schema, ensuring type safety when querying tables like `bookings`.
* **`supabase/auth-middleware.ts` / `client.server.ts`**: Utilities for handling auth states and server-side connections (used primarily if SSR or trusted server contexts are needed).

### 5. `supabase/`
Contains the backend configuration and Edge Functions.
* **`functions/send-booking-email/index.ts`**: A Deno-based serverless function. It receives a POST request from the client containing the newly booked details. It formats two HTML emails (one for the user, one for the admin) and posts them to the `api.resend.com` endpoint. It operates gracefully, returning a `200 OK` even on email failure, to prevent breaking the user's booking experience.
* **`migrations/`**: Contains the SQL files defining the schema for the application, such as creating the `bookings` table, enabling Row Level Security (RLS), and defining the `get_public_bookings` RPC function.
* **`config.toml`**: The local configuration file for the Supabase CLI.

## Key Components and Responsibilities

1. **`BookingCalendar` (`src/components/booking/BookingCalendar.tsx`)**
   * **Responsibility:** Visualizing date availability.
   * **Logic:** Computes grid cells for a given month. Determines if a date is interactive based on whether it exists in the `blocked` set or `bookings` map.

2. **`BookPage` (`src/pages/book.tsx`)**
   * **Responsibility:** Managing the user conversion funnel.
   * **Logic:** Orchestrates the transition from selecting a date -> reviewing policies -> filling the Zod-validated form -> submitting to Supabase -> displaying a success screen.

3. **`AdminDashboard` (`src/pages/admin/index.tsx`)**
   * **Responsibility:** Secure data management.
   * **Logic:** Enforces role-based access control (RBAC). Upon mount, it ensures the user has an `admin` role in the `user_roles` table. Provides a unified interface to read, delete, and block records.

4. **`send-booking-email` (`supabase/functions/send-booking-email/index.ts`)**
   * **Responsibility:** Outbound communications.
   * **Logic:** Securely utilizes the `RESEND_API_KEY` (hidden from the frontend) to construct and dispatch transactional emails using `fetch()`.
